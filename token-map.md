# Henri Lloyd — Token Map
## Design token → Stripo output reference
**Active modes: Dark Base · Desktop (primary) · Mobile (override layer)**

This file is the single reference for translating Figma local variables into Stripo email CSS and HTML.
All alias references (`{spacing.L}`, etc.) are resolved to their literal values.
**Never use CSS custom properties (`var()`) in email output — email clients strip them.**

---

## How to read this document

| Column | Meaning |
|---|---|
| Token path | Dot-separated path inside the JSON collection |
| Resolved value | Literal value after alias resolution |
| Stripo layer | **Default CSS** · **Custom CSS** · **Inline** |
| CSS target | The selector or HTML element that receives the value |
| Property | The CSS property name |

### The three Stripo layers

| Layer | Location in Stripo UI | What belongs here |
|---|---|---|
| **Default CSS** | Appearance → General Styles | **Read-only.** Generated by the General Styles panel (Default Styles = ON). Do not edit or replace it. |
| **Custom CSS** | Custom CSS panel | **All our token-driven overrides live here**: h1–h6 scale, body/link/list styles, colours, `@media` mobile blocks. Override Default CSS via normal cascade. |
| **Inline** | `style=""` on the HTML element | Block-specific padding, per-block background, one-off font exceptions |

**Cascade rule:** Inline always wins over Default/Custom CSS without `!important`. The only place `!important` is required is inside `@media` mobile blocks, to override desktop inline styles.

### 3.2 Body and utility styles

| Role | Token path | font-family | font-size | line-height | font-weight | CSS weight | layer | CSS target |
|---|---|---|---|---|---|---|---|---|
| Body copy | `type.body` | DM Sans | `18px` | `25.2px` (≈140%) | Regular | `400` | **Default CSS** | `.es-content-body p`, `li` |
| Footer / header text | `type.Footer` | DM Sans | `14px` | `19.6px` (≈140%) | Regular | `400` | **Default CSS** | `.es-footer-body p`, `.es-header-body p` |
| Info-area small print | `type.Info-area` | DM Sans | `12px` | `16.8px` (≈140%) | Regular | `400` | **Default CSS** | `.es-infoblock p` |
| Button label | `type.button` | DM Sans | `12px` | `12px` (100%) | Bold | `700` | Inline | `<a class="es-button">` |

> **Why all three paragraph roles are Default CSS — not Inline:**
> Stripo's native Default CSS (generated by General Styles with Default Styles ON) already declares `font-size` for `.es-content-body p` (18px), `.es-footer-body p` / `.es-header-body p` (14px), and `.es-infoblock p` (12px) in the correct cascade order. All three selectors share equal specificity (0-1-1). Source-order alone resolves which rule wins — `.es-infoblock p` is declared last and therefore always overrides `.es-content-body p` for any `<p>` inside an infoblock TD, regardless of DOM nesting depth.
>
> **Consequence:** The `<p>` tag itself must be kept completely clean — no `style="font-size"`, no override classes. The correct block type is communicated entirely by the class on the parent `<td>` (e.g., `es-infoblock`, `es-footer-body`). Adding inline font-size to a `<p>` breaks this cascade and forces Stripo to generate compensating noise rules in Custom CSS. Do not do it.

**`type.Info-area` font-weight:** `Regular (400)`. Aliases `Font Weight/body copy` from the Brand collection.

**Button letter-spacing:** Token value is `1px` (updated from Figma `0` to match existing `defaul.css` — confirmed decision). Applied inline on `<a class="es-button">`.

## 4. Typography tokens — Mobile

Source: `Figma-local-variables/Typography Tokens - New/Mobile.tokens.json`
Active mode: **Mobile**
Layer: **Custom CSS** — exclusively inside `@media only screen and (max-width:600px)` with `!important` on all values.

> **Mode mapping rule:**
> - `Desktop.tokens.json` → base `h1`–`h6` rules in the **Custom CSS panel** (no `@media` wrapper)
> - `Mobile.tokens.json` → all heading and body overrides inside the **`@media` block** in Custom CSS
> These two files are the only sources of truth. Never mix modes. Never apply mobile token values outside the `@media` block.

---

### 4.1 — Stripo's native responsive class anatomy

Stripo uses three classes in combination to wire every responsive text block. The AI must understand all three before generating any `@media` CSS.

| Class | Lives on | Purpose |
|---|---|---|
| `es-text-{ID}` | `<td class="esd-block-text">` | **Scope anchor.** A unique integer ID assigned by Stripo to this specific block. It namespaces the `@media` rule so it fires only on this block, not globally. |
| `es-text-mobile-size-{XX}` | Inner element (`<h1>`–`<h6>`, `<p>`) | **Size declaration.** The integer `{XX}` encodes the target mobile font-size. This class name IS the value — `es-text-mobile-size-36` means 36px. |
| `es-override-size` | Inner element (`<h1>`–`<h6>`, `<p>`) | **Enrollment flag.** Signals that this element has a confirmed responsive override. Without this class the override system is not enrolled and Stripo compensates with inline styles (noise). |

**Canonical HTML pattern — correctly enrolled responsive block:**
```html
<td align="left" class="esd-block-text es-p30b es-text-1766">
  <h1 class="es-override-size es-text-mobile-size-36" style="font-size: 48px">
    Headline copy.
  </h1>
</td>
```

> **Note on the desktop inline style:** `style="font-size: 48px"` appears on `h1` because its 48px desktop size differs from Stripo's Default CSS default for `h1`. Stripo locks it inline to prevent cascade drift. h2–h6 and `<p>` do NOT carry this inline style when their desktop size matches the Default CSS value.

---

### 4.2 — Alias resolution rule for line-height

Many heading `line-height` tokens in both JSON files use a self-referencing alias:
```json
"line-height": { "$value": "{type.headings.h1.font-size}" }
```
This means **line-height equals the font-size of the same element in the same mode file**. Resolution is always per-mode:
- In `Desktop.tokens.json`: h1 font-size = 48 → line-height = 48px
- In `Mobile.tokens.json`: h1 font-size = 36 → line-height = 36px

Never cross-reference modes when resolving aliases. `h5` and `type.body` use explicit numeric line-height values — no alias to resolve.

### 4.4 — Strict CSS template for mobile override generation

The AI **must** use this exact template structure for every heading element that requires a mobile override. No deviation permitted.

```css
/* TEMPLATE — one block per heading element */
.es-text-{ID} .es-text-mobile-size-{MOBILE_SIZE}.es-override-size,
.es-text-{ID} .es-text-mobile-size-{MOBILE_SIZE}.es-override-size * {
  font-size: {MOBILE_SIZE}px !important;
  line-height: {MOBILE_LINE_HEIGHT}px !important;
}
```

- `{ID}` — read from the `es-text-{ID}` class already on the `<td>` in the HTML
- `{MOBILE_SIZE}` — read from `Mobile.tokens.json` → `type.headings.hX.font-size`
- `{MOBILE_LINE_HEIGHT}` — read from `Mobile.tokens.json` → `type.headings.hX.line-height` (alias-resolved within the mobile file)
- The `*` descendant selector is mandatory — it covers inline `<a>`, `<strong>`, `<span>` children
- `!important` is mandatory on both properties — it must override the desktop inline style (`style="font-size: 48px"`) that Stripo locks on h1

### 4.5 — Complete `@media` block output (resolved from current tokens)

```css
@media only screen and (max-width: 600px) {

  /* h1: 48px → 36px | IDs from example.html reference */
  .es-text-1766 .es-text-mobile-size-36.es-override-size,
  .es-text-1766 .es-text-mobile-size-36.es-override-size * {
    font-size: 36px !important;
    line-height: 36px !important;
  }

  /* h2: 32px → 28px */
  .es-text-1355 .es-text-mobile-size-28.es-override-size,
  .es-text-1355 .es-text-mobile-size-28.es-override-size * {
    font-size: 28px !important;
    line-height: 28px !important;
  }

  /* h3: 24px → 22px */
  .es-text-2007 .es-text-mobile-size-22.es-override-size,
  .es-text-2007 .es-text-mobile-size-22.es-override-size * {
    font-size: 22px !important;
    line-height: 22px !important;
  }

  /* h4: 20px → 18px */
  .es-text-2609 .es-text-mobile-size-18.es-override-size,
  .es-text-2609 .es-text-mobile-size-18.es-override-size * {
    font-size: 18px !important;
    line-height: 18px !important;
  }

  /* h5: 15px → 14px */
  .es-text-2513 .es-text-mobile-size-14.es-override-size,
  .es-text-2513 .es-text-mobile-size-14.es-override-size * {
    font-size: 14px !important;
    line-height: 16.8px !important;
  }

  /* h6: 12px → 10px */
  .es-text-8291 .es-text-mobile-size-10.es-override-size,
  .es-text-8291 .es-text-mobile-size-10.es-override-size * {
    font-size: 10px !important;
    line-height: 10px !important;
  }

  /* body p: 18px → 16px | Zone-wide — no block ID needed */
  .es-content-body p,
  .es-content-body a {
    font-size: 16px !important;
    line-height: 24px !important;
  }

  /* CASCADE PROTECTION — must be written AFTER the content-body rule above.
     type.Footer (14px) and type.Info-area (12px) are desktop = mobile, so no
     token-driven size change is needed. However, .es-content-body p above would
     bleed 16px into footer and infoblock zones without these guards.
     Both rules have equal specificity + !important — source order decides.
     type.button produces no @media output (desktop = mobile, applied inline). */
  .es-footer-body p,
  .es-footer-body a {
    font-size: 14px !important;
    line-height: 19.6px !important;
  }

  .es-infoblock p,
  .es-infoblock a {
    font-size: 12px !important;
    line-height: 16.8px !important;
  }

}
```

> **ID notice:** The `es-text-{ID}` values above (1766, 1355, 2007, etc.) are from `example.html` and are used as the canonical reference. Each new template import into Stripo will generate its own IDs. The AI must read the actual IDs from the target HTML before generating CSS — never hardcode IDs from the reference file into a new template's CSS.

## 10. AI Pipeline Execution Rules

These rules are **mandatory for any AI agent generating Stripo HTML or CSS output** for this project. They encode the architectural decisions confirmed by analysis of `original-defaul.css` and `example.html`. Violations corrupt the native cascade and produce unmaintainable output.

---

### 10.1 — Never inject inline font-size or line-height on base `<p>` tags

**FORBIDDEN:**
```html
<p style="font-size: 12px; line-height: 16.8px">...</p>
<p class="es-text-mobile-size-12" style="font-size: 12px">...</p>
```

**CORRECT:**
```html
<p>...</p>
```

The `<p>` element must carry **no `style` attribute** and **no font-size or line-height class**. The font-size is already resolved by Stripo's Default CSS via the parent TD's class. There is nothing to add.

This applies to all paragraph roles without exception:

| Role | font-size resolved by |
|---|---|
| Body copy (18px) | `.es-content-body p` in Default CSS |
| Footer / header text (14px) | `.es-footer-body p` / `.es-header-body p` in Default CSS |
| Info-area small print (12px) | `.es-infoblock p` in Default CSS |

---

### 10.2 — Apply block type via the parent `<td>`, never via the `<p>`

The sole mechanism for telling Stripo which paragraph size to render is the class on the **wrapping `<td>`**. The AI must place the correct zone/utility class there and nowhere else.

**Info-area pattern (renders at 12px automatically):**
```html
<td align="left" class="esd-block-text es-infoblock">
  <p>Small print copy goes here.</p>
</td>
```

**Footer pattern (renders at 14px automatically):**
```html
<!-- The parent stripe table already carries es-footer-body.
     No additional class needed on the TD beyond esd-block-text. -->
<td align="left" class="esd-block-text">
  <p>Footer copy goes here.</p>
</td>
```

**Body copy pattern (renders at 18px automatically):**
```html
<td align="left" class="esd-block-text">
  <p>Body copy goes here.</p>
</td>
```

---

### 10.3 — Custom CSS is for H1–H6 and mobile `@media` only

The Custom CSS panel must **never** contain:

- `.es-content-body p { font-size: ... }` — this redeclaration fights Default CSS
- `.es-infoblock p { font-size: ... }` — this duplicates a rule already in Default CSS
- `.es-footer-body p { font-size: ... }` — same; already owned by Default CSS
- Any bare `p { font-size: ... }` rule

**What Custom CSS is allowed to contain:**

| Allowed in Custom CSS | Reason |
|---|---|
| `h1` – `h6` selector rules (font-size, line-height, font-weight, margin-bottom, letter-spacing, color) | Headings are token-driven and not fully covered by Default CSS out of the box |
| `@media only screen and (max-width:600px)` blocks | Mobile overrides require `!important` to beat desktop inline styles; they belong in Custom CSS |
| Per-module scoped overrides (`.esd-module-{id}`) | When a single block needs a one-off heading or spacing exception |
| Mobile heading size overrides (h1–h6 inside `@media`) | Desktop heading sizes change on mobile; Custom CSS owns this layer |
| Mobile body copy override (`font-size: 16px !important` on `.es-content-body p, .es-content-body a`) | Body copy drops from 18px → 16px on mobile; this is the one paragraph-level `@media` override that is permitted |

**The required mobile paragraph block (content-body reduction + cascade protection guards):**
```css
@media only screen and (max-width: 600px) {
  /* 1. Reduce body copy: 18px → 16px */
  .es-content-body p,
  .es-content-body a {
    font-size: 16px !important;
    line-height: 24px !important;
  }
  /* 2. Protect footer — written after, wins by source order */
  .es-footer-body p,
  .es-footer-body a {
    font-size: 14px !important;
    line-height: 19.6px !important;
  }
  /* 3. Protect infoblock — written last, wins over both */
  .es-infoblock p,
  .es-infoblock a {
    font-size: 12px !important;
    line-height: 16.8px !important;
  }
}
```

> **Why the guards are mandatory:** `.es-footer-body` and `.es-infoblock` are nested inside `.es-content-body` in the DOM. The `!important` body rule at (0,1,1) would therefore bleed into footer and infoblock paragraphs. Adding the guard rules at the same specificity but later in source order reclaims the correct values. See §10.8 for full diagnosis.

---

### 10.4 — The cascade contract (do not break any layer)

```
Default CSS (Stripo-generated, read-only)
  └── owns: all base p font-sizes, wrapper backgrounds, font-family, list styles
        ↓ overridden by cascade / source-order for zone-specific rules

Custom CSS (our token overrides, editable)
  └── owns: h1–h6 scale, mobile @media blocks
        ↓ overridden by specificity

Inline styles (block-specific only)
  └── owns: per-block padding, per-block background, button colours
  └── NEVER owns: p font-size, p line-height, h font-size (those belong above)
```

Breaking this contract at any layer causes one or more of:
- Stripo UI showing conflicting values ("overridden in Code editor")
- The editor generating compensating noise rules (`.es-text-{id}` clutter)
- Mobile `!important` overrides failing to fire because inline styles win specificity
- Visual regressions when the template is re-edited in Stripo's visual UI

---

### 10.5 — HTML class immutability — structural classes are read-only; `es-text-mobile-size-{XX}` is the sole exception

All `es-` and `esd-` classes in the HTML are **read-only** to the AI pipeline with one strict exception.

**Classes the AI must never add, rename, or remove:**
- `esd-stripe`, `esd-structure`, `esd-container-frame`, `esd-block-*` — structural wiring
- `es-override-size` — Stripo enrollment flag; do not add or remove it
- `es-text-{ID}` on any `<td>` — Stripo-assigned scope anchor; never invent or change
- Any class on a `<p>` element — font-size is resolved by the parent TD's zone class, not the `<p>` itself

**The one permitted HTML class change — `es-text-mobile-size-{XX}` sync:**

The integer suffix `{XX}` in `es-text-mobile-size-{XX}` encodes the mobile font-size. It must be identical to the `font-size` value in the matching `@media` CSS rule. Stripo uses this class to keep the visual editor and the compiled CSS in sync. If they diverge, the template is silently broken in Stripo's UI.

**Sync rule:** When applying Figma Mobile tokens, the AI **must** compare the existing `es-text-mobile-size-{XX}` class value against the token's `font-size`. If they differ, the AI must update the HTML class to match the token before generating the CSS. Both changes are made together as one atomic operation.

**Example — token says 22px, HTML has the old value 24:**
```html
<!-- BEFORE (stale, from previous Stripo session) -->
<h3 class="es-override-size es-text-mobile-size-24">Headline</h3>

<!-- AFTER (synced to Figma Mobile token) -->
<h3 class="es-override-size es-text-mobile-size-22">Headline</h3>
```
```css
/* CSS generated after the HTML is synced */
.es-text-2007 .es-text-mobile-size-22.es-override-size,
.es-text-2007 .es-text-mobile-size-22.es-override-size * {
  font-size: 22px !important;
  line-height: 22px !important;
}
```

The class name suffix, the CSS selector, and the `font-size` value are always the same integer. There is no valid state where they differ.

---

### 10.6 — How to read the target HTML before generating `@media` CSS

Before writing any `@media` block for a new template, the AI must perform this read sequence on the target HTML file:

1. **Find every `es-text-{ID}` class** on `<td>` elements → these are the scope anchors
2. **Find the inner element** of each such TD — record which tag it is (`h1`–`h6` or `p`)
3. **Record `es-text-mobile-size-{XX}`** from the inner element → this is the mobile size already declared in the HTML
4. **Confirm `es-override-size` is present** on the inner element → if absent, flag the block as not enrolled (do not generate a `@media` rule for it; flag it for human review)
5. **Match each block** to the corresponding Figma Mobile token for its element type → extract the token's `font-size` and `line-height`
6. **Sync the HTML class if needed** — compare the `{XX}` suffix of `es-text-mobile-size-{XX}` against the token's `font-size`. If they differ, update the HTML class to the token value before writing any CSS.
7. **Generate the `@media` rule** using the (now-synced) token `font-size` value as both the selector suffix and the `font-size` property. Use the token's `line-height` for the `line-height` property.

This sequence guarantees the CSS selector, the HTML class, and the token value are always the same integer — which is the only valid state.

### 10.8 — Footer and infoblock cascade protection in `@media`

**Root cause:** `.es-footer-body` and `.es-infoblock` are nested *inside* `.es-content-body` in the DOM. When the `@media` block reduces `.es-content-body p` to `16px !important`, that rule applies to every `<p>` inside any content stripe — including footer and infoblock paragraphs. Both rules have identical specificity (0,1,1) and both carry `!important`. CSS tie-breaking rule: **last declaration in source order wins**. Because Custom CSS loads after Default CSS, the 16px body rule silently overwrites the 14px and 12px values that Default CSS set.

**Two conditions that must both be true to avoid breakage:**

**Condition 1 — Clean `<p>` HTML in zone blocks.**
`<p>` tags inside `es-infoblock` and `es-footer-body` TDs must carry **no** mobile size classes. If `es-text-mobile-size-{XX}` or `es-override-size` appear on a zone `<p>`, Stripo treats the block as enrolled in the heading-style responsive system and injects its own override — which then fights the cascade protection rules below.

```html
<!-- FORBIDDEN inside es-infoblock or es-footer-body -->
<td class="esd-block-text es-infoblock">
  <p class="es-text-mobile-size-16 es-override-size">Copy here.</p>
</td>

<!-- CORRECT — clean <p>, zone resolved by parent TD class -->
<td class="esd-block-text es-infoblock">
  <p>Copy here.</p>
</td>
```

**Condition 2 — Cascade protection rules, in the correct source order.**
The `@media` block must always contain these three rules in this exact order. Do not reorder them. Do not omit the footer or infoblock guards.

```css
@media only screen and (max-width: 600px) {

  /* 1. Body copy reduction — triggers the cascade problem */
  .es-content-body p,
  .es-content-body a {
    font-size: 16px !important;
    line-height: 24px !important;
  }

  /* 2. Footer guard — written after body rule; reclaims 14px by source order */
  .es-footer-body p,
  .es-footer-body a {
    font-size: 14px !important;
    line-height: 19.6px !important;
  }

  /* 3. Infoblock guard — written last; wins over both rules above */
  .es-infoblock p,
  .es-infoblock a {
    font-size: 12px !important;
    line-height: 16.8px !important;
  }

}
```

**Why these values:** `type.Footer` is 14px/19.6px and `type.Info-area` is 12px/16.8px on both desktop and mobile (no token delta). The guards are not size-change rules — they are shield rules that re-assert the Default CSS values inside the Custom CSS `@media` block to prevent source-order bleed from the content-body reduction.

**Cascade diagram:**
```
Default CSS (loaded first)
  .es-infoblock p { font-size: 12px }         ← set by Default CSS

Custom CSS @media (loaded after — later source order)
  .es-content-body p { font-size: 16px !important }  ← overwrites infoblock ← BAD without guard
  .es-footer-body p  { font-size: 14px !important }  ← reclaims footer
  .es-infoblock p    { font-size: 12px !important }  ← reclaims infoblock ← CORRECT
```

Before finalising any HTML or CSS output, verify:

**HTML hygiene:**
- [ ] All `<p>` tags inside `esd-block-text` TDs are attribute-free (no `style`, no font-size class)

- [ ] The `*` descendant selector is appended on the second line of each rule
- [ ] `!important` is present on both `font-size` and `line-height`

- [ ] No token-vs-HTML size mismatch remains in the output — mismatches are resolved by updating the HTML class, not by leaving them flagged

### 10.9 — The Mobile Cascade Protection Rule (canonical reference)

> **Full specification:** `stripo-rules.md §9`. This section is the token-map summary. Always read §9 of stripo-rules.md for the authoritative rule before generating any `@media` block that touches body copy.

**Trigger condition:** Anytime `.es-content-body p` or `.es-content-body a` is resized inside a `@media` block, the AI **must** write all three of the following rules, in this exact order, immediately after the body rule:

```css
@media only screen and (max-width: 600px) {

  /* 1. Body override */
  .es-content-body p,
  .es-content-body a {
    font-size: [Mobile_Body_Size] !important;
    line-height: [Mobile_Body_LH] !important;
  }

  /* 2. Footer protection — must follow rule 1 directly */
  .es-footer-body p,
  .es-footer-body a {
    font-size: [Desktop_Footer_Size] !important;
    line-height: [Desktop_Footer_LH] !important;
  }

  /* 3. Infoblock protection — must be last; wins by source order */
  .es-infoblock p,
  .es-infoblock a {
    font-size: [Desktop_Infoblock_Size] !important;
    line-height: [Desktop_Infoblock_LH] !important;
  }

}
```

**Resolved values (Henri Lloyd token set):**

| Placeholder | Token | Value |
|---|---|---|
| `[Mobile_Body_Size]` | `type.body` Mobile | `16px` |
| `[Mobile_Body_LH]` | `type.body.line-height` Mobile | `24px` |
| `[Desktop_Footer_Size]` | `type.Footer` | `14px` |
| `[Desktop_Footer_LH]` | `type.Footer.line-height` | `19.6px` |
| `[Desktop_Infoblock_Size]` | `type.Info-area` | `12px` |
| `[Desktop_Infoblock_LH]` | `type.Info-area.line-height` | `16.8px` |

**Root cause in one sentence:** `.es-footer-body` and `.es-infoblock` are DOM children of `.es-content-body`; all three rules share equal specificity (0,1,1) + `!important`; source order decides — body rule (written first) bleeds into footer and infoblock unless guards follow immediately after it.

**Three non-negotiable requirements:**
1. Source order is fixed: rule 1 → rule 2 → rule 3. Do not reorder.
2. All three rules are written together. Rule 1 alone is a pipeline error.
3. `<p>` tags inside `es-infoblock` and `es-footer-body` TDs must carry no `es-text-mobile-size-{XX}` or `es-override-size` class — zone size is resolved by the parent TD class, not the `<p>` itself.
